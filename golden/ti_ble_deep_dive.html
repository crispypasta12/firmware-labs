LAB 22The BLE protocol stack — and what each layer adds to a packetArchitecture
Interview angle: "walk me through the BLE stack" is the opener for any connectivity role. Click layers for their job, then press Send a notification and watch the packet grow headers as it descends — the byte-level view is what makes the answer senior.

APPLICATION / PROFILES HOST

Your code + SIG profiles (Heart Rate, Environmental Sensing…). Decides what to send and when. On TI: your app task talking to the BLE5-Stack via ICall messages.

GAP · GATT HOST

GAP: roles (central / peripheral / broadcaster / observer), advertising, connecting, security modes. GATT: the data model — services, characteristics, descriptors — built on ATT. GATT Server holds data; GATT Client reads/writes/subscribes.

ATT · SMP HOST

ATT: the wire protocol — opcodes like Read Request (0x0A), Write Request (0x12), Handle Value Notification (0x1B). One outstanding transaction at a time, 30 s ATT timeout = disconnect. SMP: pairing, key generation/distribution.

L2CAP HOST

Multiplexing + fragmentation. Fixed channels: CID 0x0004 = ATT, 0x0006 = SMP, 0x0005 = signaling (connection parameter update request lives here). CoC (connection-oriented channels) give credit-based flow control for high-throughput streams (used by OTA on some stacks).

HCI BOUNDARY

Standardized command/event interface between Host and Controller. On single-chip designs (CC26xx/CC23xx) it's an internal API; on dual-chip (network processor) it runs over UART/SPI. Useful in debugging: HCI logs show exactly what the host asked the controller to do.

LINK LAYER CONTROLLER

The hard real-time core: advertising/scanning/connection state machines, channel hopping, encryption (AES-CCM), LL control procedures (connection update, channel map update, PHY update, LL_TERMINATE_IND). On CC26xx this runs on the dedicated M0 RF core.

PHY CONTROLLER

2.4 GHz ISM, GFSK, 40 channels × 2 MHz. 1M (mandatory), 2M (throughput), Coded S=2/S=8 (range, −4/−12 dB sensitivity gain). TX power, receiver sensitivity (~−97 dBm @1M on CC2340) set the link budget.

LAB 2340 channels & Wi-Fi coexistence — why 37/38/39 live where they doPHY / RF
Interview angle: "BLE works on the bench but fails in the office" → 2.4 GHz interference. Toggle the Wi-Fi networks and see which BLE channels get stomped — and notice the three advertising channels sitting precisely in the gaps.
Hover or tap a channel. Purple = advertising (37, 38, 39) · teal = data (0–36).
Facts to drop: 40 channels, 2 MHz spacing, 2402–2480 MHz. Advertising channels 37 (2402), 38 (2426), 39 (2480) were deliberately placed in the gaps between Wi-Fi 1/6/11 so discovery survives even in dirty RF. Data channels that overlap an access point get excluded by the host via the channel map / adaptive frequency hopping (next lab).

The recipe to recite for fast OTA: negotiate MTU 247 + Data Length Extension (LL payload 27→251 B, BLE 4.2) + 2M PHY + short connection interval + notifications/Write-Without-Response with app-level ACKs every N chunks (not indications — one round trip each). LAB 27Pairing & security — the IO-capability matrixSMP
Interview angle: "your device has no display or buttons — how does pairing work, and what's the risk?" Pick the IO capabilities and read the resulting method straight off. Then say the LESC sentence below.

—

The LESC sentence: "Legacy pairing derives keys from a TK that Just-Works sets to zero — passively sniffable. LE Secure Connections does an ECDH P-256 exchange, so even Just Works resists passive eavesdropping; it just can't stop an active MITM without numeric comparison, passkey, or OOB." Pairing = creating keys for this session; bonding = storing them (LTK to encrypt reconnects, IRK to resolve private addresses, CSRK for signed writes). BLE link encryption itself is AES-128-CCM in the Link Layer.

Why firmware engineers care: if you whitelist/filter by address without storing the IRK, reconnection breaks the moment the phone rotates its RPA — a very common field bug. Controllers resolve RPAs in hardware against the resolving list. RPA = 24-bit prand (top bits 01) + 24-bit hash ah(IRK, prand); the peer recomputes the hash with each stored IRK and compares.
LAB 29Link Layer state machine — drive itLink Layer
Interview angle: roles fall out of this machine: enter Connection from Advertising and you're the peripheral; from Initiating and you're the central. Drive both routes.
GAP roles map: Broadcaster = advertising only (beacon). Observer = scanning only. Peripheral = advertise → accept connection. Central = scan → initiate → own the connection (decides parameters, distributes the channel map). One device can hold multiple roles at once on modern stacks — e.g. a CC2652 can be peripheral to a phone while central to a sensor.
LAB 30BLE 4.0 → 5.4 — what arrived whenSpec literacy
Interview angle: dating features ("DLE is 4.2, 2M PHY is 5.0, subrating is 5.3") instantly signals depth. Click a version. Note: CC2340 ships BLE 5.3 — connection subrating is its headline.
Two you must own for TI: Connection Subrating (5.3) — a connected device skips most events by a negotiated factor but can ramp to full rate instantly; better than slave latency for "idle but must respond fast" products (earbuds, locks). PAwR (5.4) — periodic advertising with responses; thousands of nodes synchronized to one broadcaster — electronic shelf labels.
LAB 31Senior debugging playbook — symptom → causeField experience
Interview angle: these are the bugs that actually happen. Quiz yourself: read the symptom, name causes out loud, then expand and compare. Tooling baseline: sniffer (TI Packet Sniffer 2 / nRF Sniffer + Wireshark), HCI logs, RSSI, and a spectrum view.
The sniffer habit: capture over-the-air first, then look at LL_TERMINATE_IND — it tells you which side dropped the link and the reason code (0x08 supervision timeout, 0x13 remote user terminated, 0x16 local host terminated, 0x3D MIC failure = encryption/key mismatch). Half of all BLE debugging is knowing who hung up and why.

TI BLE5-Stack quick reference (CC26xx) — sprinkle these in

  • Architecture: stack runs as its own task above the RF driver; your app exchanges ICall messages with it. On CC23xx the stack moves toward a more library-style integration, and TrustZone can isolate the controller.
  • GATT server: services registered with GATTServApp_RegisterService(); each service supplies an attribute table + read/write callbacks. Newer SDKs generate much of this from SysConfig.
  • GAP: advertising sets via GapAdv_*, connection handling via GapInit_*/GAP events; bond storage in NV via GapBondMgr (stores LTK/IRK — the RPA lab is why).
  • Power: the power driver + RF core let the M4 sleep through connection events automatically — you mostly just avoid holding constraints (Power_setConstraint) longer than needed.
  • Tools: SimpleLink Starter / SysConfig for config, TI Packet Sniffer 2 + Wireshark for OTA capture, EnergyTrace for current profiling of connection parameters.
