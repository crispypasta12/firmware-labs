BRR = f_periph / baud. Recomputed live as you change the tree — that's the correct behavior.
The answer pattern: "Baud is derived from the peripheral clock through BRR. Changing SYSCLK or a bus divider changes f_periph but BRR keeps its old value → actual baud diverges → framing errors / garbage. Fix: recalculate BRR after any clock change, and check which APB bus the UART hangs off in the clock tree." UART tolerates roughly ±2% baud error.
LAB 17I2C write-read transaction — decoded waveformPDF Q43 · Hub S1
Interview angle: "debug an I2C failure with a logic analyzer" — this is what you'd see on screen. Hover/tap any region for what it means. Reading a register at slave 0x48 (a TMP117-style temp sensor): write the register address, repeated start, read the data.
address + R/W register data (slave drives) start / rep-start / stop NACK
Hover or tap a region of the waveform.
Logic-analyzer checklist: no ACK on address → wrong address / dead / unpowered slave. Slow rising edges → wrong pull-ups (setup violation). SCL held low forever → stretch-timeout or stuck slave → bus recovery: toggle SCL ~9×, send STOP, reinit. Master must change SDA only while SCL is low — except START (fall) and STOP (rise) which happen while SCL is high; that's exactly what makes them unique.

LAB 19Thread mesh — sleepy end devices & self-healing routesHub S8 · Matter context
Interview angle: TI's CC13xx/CC26xx push Thread + Matter hard. Deliver a message to a sleeping sensor, then kill a router and watch the mesh route around it — the two behaviors that define Thread.
BR = border router (bridges to IP/Wi-Fi). R1/R2 = always-on routers. R3 = the SED's parent. SED = sleepy end device — radio OFF between polls.
SED in one breath: "It doesn't route and isn't always-on — it polls its parent every few seconds; the parent buffers downstream messages; the mesh routes around it. That's how a CC2652 sensor runs for years on a battery. Contrast: MTD polls faster, FTD is always-on and routes." Matter then runs over Thread (or Wi-Fi/Ethernet) as the application layer, with the border router bridging to IP.

Interview angle: "what is a context switch?" answered at register level is a senior flex. Watch which frame the hardware pushes vs which the kernel pushes, and where MSP vs PSP is in use.

Three facts to drop: ① hardware auto-pushes xPSR·PC·LR·R12·R3–R0 on exception entry; the kernel (PendSV) saves only R4–R11 + PSP. ② Handlers run on MSP; tasks run on PSP — that's why each task needs its own stack sized for its worst case. ③ The switch happens in PendSV at the lowest priority, so a context switch can never preempt another ISR mid-flight.
LAB 21RMS schedulability quick-checkHub S4
Interview angle: if scheduling theory comes up, doing the Liu & Layland check mentally is impressive. Practice with a few sets here.
Task CPU utilizations (%):
Recite: RMS = shorter period → higher static priority; optimal among fixed-priority schemes. Bound: U ≤ n(21/n − 1) → 69.3% as n→∞. Under the bound = guaranteed. Between bound and 100% = maybe (needs response-time analysis). Over 100% = never.
