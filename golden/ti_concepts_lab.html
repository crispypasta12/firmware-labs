LAB 03Ring buffer — head, tail, full vs emptyPDF Q20 · Q47
Interview angle: "design a UART RX path" almost always lands here. The trap question: how do you tell full from empty when head == tail? (Answer: leave one slot unused, or keep a count.)
head (write): 0 tail (read): 0 count: 0 state: EMPTY overruns dropped: 0
ISR/DMA writes at head. Task reads at tail. One slot is kept unused so full ≠ empty.
Conditions to recite: empty when head == tail; full when (head+1) % N == tail. Indexes shared between ISR and task must be volatile and updated single-writer (ISR owns head, task owns tail) — that's what makes the lock-free version safe.
LAB 04Circular DMA + half/full interrupts (ping-pong)PDF Q8–Q9 · Q45
Interview angle: your CV claims circular DMA on UART/ADC — expect "walk me through it" and "what goes wrong?" Toggle the slow CPU and watch the overrun happen: that's the failure mode to name unprompted.
← first half · HT interrupt fires when full
second half · TC interrupt fires, DMA wraps to 0 →
HT events: 0 TC events: 0 overruns: 0
DMA fills cells (amber). On HT/TC interrupt, the CPU consumes that half (teal). Press Play.
The rule: ping-pong only works if processing time per half < fill time per half. Bugs to list: wrong address/length/width, missed flag clear, ISR/task index races, wraparound math, and cache coherency on MCUs with D-cache.

LAB 06HardFault detective — a worked investigationPDF Q29 · Q49
Interview angle: "how do you debug a HardFault?" separates juniors from seniors. The senior answer names the registers. Walk this real-style case until the register names are automatic.
▸ Symptom
Device resets randomly under load. You attach the debugger and halt — PC is inside HardFault_Handler. Don't guess. Read the fault registers.
▸ HFSR @ 0xE000ED2C = 0x4000_0000
Bit 30 FORCED = 1 → a configurable fault (bus/usage/mem) escalated to HardFault because its handler wasn't enabled. So the real story is in CFSR.
▸ CFSR @ 0xE000ED28 = 0x0000_8200
Bus fault byte: PRECISERR = 1 (precise data bus error — the stacked PC points at the exact faulting instruction) and BFARVALID = 1 → the faulting address is captured in BFAR.
▸ BFAR @ 0xE000ED38 = 0x0000_0004
Faulting address is near zero → a NULL pointer dereference (reading a struct field at offset 4 through a NULL pointer).
▸ Stacked PC (from exception stack frame) = 0x0000_A4C2
The CPU auto-pushed R0–R3, R12, LR, PC, xPSR on fault entry. Look up 0xA4C2 in the .map file / disassembly → it lands inside uart_send(), at a load through the driver handle.
▸ Root cause
A task called uart_send() before uart_init() ran — the handle pointer was still NULL. Fix: assert non-NULL handle at entry, fix init ordering, add an init-state flag. Process change: add an assertion-on-entry convention for all driver handles.
Press "Next clue" to start the investigation.
Register cheat line: CFSR 0xE000ED28 (which sub-fault) → HFSR 0xE000ED2C (escalated?) → MMFAR/BFAR 0xE000ED34/38 (faulting address) → stacked PC (faulting instruction) → map file (which function). Causes to list: NULL pointer, stack overflow, bad function pointer, invalid ISR priority calling RTOS APIs, DMA writing out of bounds.

LAB 09SPI modes — CPOL / CPHA explorerPDF Q42 · Q48
Interview angle: "SPI returns garbage — first thing you check?" → mode mismatch. Click through all four modes and note which edge samples the data.
Memory hook: CPOL = where the clock rests (0 = low). CPHA = which edge samples (0 = first edge, 1 = second). Mode 0 and Mode 3 cover most devices — always confirm against the slave's datasheet timing diagram.
LAB 10App / HAL / Driver / BSP — who owns whatPDF Q6
Interview angle: this question maps directly onto your Milwaukee Tool BSP work — answer it with the layer names and a concrete example from your platform. Click each layer.

APPLICATION ▾

Owns: product behavior, business logic, state machines. Must not know: register addresses, clock trees, DMA channel numbers. Example: "send telemetry every 10 s" — calls uart_write(), never USART1->TDR.

HAL ▾

Owns: a common interface over hardware-specific implementations. Example: the same uart_write() works whether the MCU underneath is STM32 or Silicon Labs — exactly how your multi-platform firmware stayed portable.

DRIVER ▾

Owns: one peripheral or device — config, TX/RX, interrupts, DMA, error flags, callbacks. Example: UART driver handling baud, ORE/FE/PE flags, ring buffer; a sensor driver doing I2C register reads.

BSP ▾

Owns: everything board-specific — pin muxing, clock setup, power sequencing, memory layout, interrupt mapping, board init. Example: bringing up a new board revision = BSP changes only; drivers and app untouched.

HARDWARE ▾

Registers, silicon, errata. Zephyr's equivalent split: Devicetree describes this layer, Kconfig selects what software above it gets compiled in, the driver model binds them.
The senior framing: "The goal wasn't making one board work — it was separating board-specific config from generic driver logic so multiple platform variants shared the same firmware components."
LAB 11OTA dual-bank update — run the state machinePDF Q65–Q66
Interview angle: drive the update yourself — including the failure paths. Being fluent in the rollback path is what makes the answer senior. Try: start → a CRC failure → finish → crash on test boot.
Checklist to recite: chunked transfer with resume (offsets, not restart-from-zero) → per-chunk CRC → full-image SHA-256 + ECDSA signature → mark pending → test boot with confirm-or-rollback (watchdog-guarded) → anti-rollback version counter in OTP/CCFG. In Zephyr this is MCUboot — know its swap modes.
